# CLAUDE.md — Apache Polaris Component

## Chart

`polaris/polaris` from `https://downloads.apache.org/polaris/helm-chart`.
Always use the `-incubating` suffix: `--version 1.3.0-incubating`.
The suffix marks it as a pre-release — Helm will NOT find it without the explicit `--version`.

## Auth Flow

1. Client calls `POST /api/catalog/v1/oauth/tokens` with `client_credentials` grant
2. Polaris validates against principal credentials in its PostgreSQL DB
3. Returns JWT token
4. Client uses token in `Authorization: Bearer <token>` header for all subsequent calls

The default bootstrap creates realm `POLARIS` with principal `root` / secret `s3cr3t`.

`Polaris-Realm: POLARIS` header routing: required in standard mode. With
`polaris.realm-context.require-header: "false"` Polaris defaults to the first configured realm —
safe when there is only one realm (our setup). This is necessary because Trino 480 removed
`iceberg.rest-catalog.additional-header.*` support.

## Storage Types

The catalog `storageConfigInfo.storageType` sets validation policy, not actual I/O routing.

**Critical gotcha — `s3.endpoint` in storageConfigInfo is silently ignored.**
Polaris's internal S3FileIO (used for data file access) reads the endpoint from
`AWS_ENDPOINT_URL_S3` env var (AWS SDK v2 standard), NOT from `s3.endpoint` in the catalog config.

To route Polaris S3 calls to Ozone:
```yaml
extraEnv:
  - name: AWS_ENDPOINT_URL_S3
    value: "http://ozone-s3g-rest.dwh.svc.cluster.local:9878"
  - name: AWS_REGION
    value: "us-east-1"
  - name: AWS_EC2_METADATA_DISABLED
    value: "true"   # prevents SDK from trying EC2 IMDS
```

`ALLOW_INSECURE_STORAGE_TYPES: true` and `polaris.readiness.ignore-severe-issues: "true"` are
required for FILE storage (Polaris flags FILE as insecure by default).

## Bootstrap Process

The Polaris admin tool (`apache/polaris-admin-tool:1.3.0-incubating`) creates the realm and
principal. It connects directly to PostgreSQL (not through the Polaris REST API). Must run AFTER
the PostgreSQL cluster is ready and the `polaris-persistence` secret exists.

Run via `kubectl run --rm -i` (one-shot pod) — not a Job, to avoid retry issues.

## RSA Token Broker Secret

The `polaris-token-broker` secret contains:
- `private.pem` — RSA private key for signing JWTs
- `public.pem` — RSA public key for verifying JWTs
- `symmetric.key` — legacy key (unused in RSA mode, but required by chart)

Generated by `install.sh` using `openssl genrsa` + `openssl rsa`.

## Polaris REST API vs Management API

Two separate API paths — never mix them:
- `/api/catalog/v1/` — **Iceberg REST Catalog spec** (Spark, Trino, PyIceberg)
- `/api/management/v1/` — **Polaris admin API** (catalogs, namespaces, principals, roles)

OAuth token endpoint is under the **catalog** path: `/api/catalog/v1/oauth/tokens`.

## Credential Vending (X-Iceberg-Access-Delegation)

PyIceberg 0.11.1 automatically adds `X-Iceberg-Access-Delegation: vended-credentials` to
`load_table` requests. Polaris tries to vend credentials via STS — but in this setup STS is
unavailable (`stsUnavailable: true` in storageConfigInfo).

Error: `Credential vending was requested for table bronze.orders, but no credentials are available`

Fix: suppress delegation in PyIceberg catalog config:
```python
catalog = load_catalog("lakehouse", **{
    "type": "rest",
    ...
    "header.X-Iceberg-Access-Delegation": "",   # suppress vended-credentials
})
```

## NodePort

Polaris service on NodePort `30181`. Accessible at `http://<node-ip>:30181`.
Internal service: `polaris.dwh.svc.cluster.local:8181`.
